# 酒店订单详情页设计

## 页面定位

酒店订单详情页当前页面名称统一为：**订单详情**。

结合现有 `pages/detail/index.html` 原型，它已经不是旧文档中描述的“左侧任务切换 + 右侧详情联动”的三段式结构，而是一个更完整的**订单履约总览页**，主要承接：

- 订单基础识别
- SLA 与风险提示
- 订单与入住信息查看
- 采购尝试过程查看
- 订单轨迹回放
- 金额与状态汇总
- 底部业务动作

同时，当前模块还存在一个被列表页抽屉承载的 `side-drawer.html` 版本，因此详情能力实际上分为：

1. 独立详情页原型：`pages/detail/index.html`
2. 抽屉详情页原型：`pages/detail/side-drawer.html`

本文以当前主页面 `index.html` 为主，必要时补充抽屉版差异。

3. 左侧主内容区
   - 订单与入住信息
   - 采购单信息
   - 订单轨迹
4. 右侧摘要区
   - 金额与结算
   - 状态摘要
5. 底部固定操作栏
6. 失败原因 / 支付补充等弹窗

说明：

- 当前页面**已经有**顶部总览区、展开面板、底部固定操作栏和多个业务按钮

### 2. 顶部头部区

### 5. 采购单信息区

当前详情页不再用“左侧任务卡片切换”方式展示采购，而是采用**按日期分组的采购单时间线/列表**。

#### 5.1 当前结构

采购单信息区按天分组展示，例如：

- 2026-04-22
- 2026-04-23
- 2026-04-24

每个日期块下可包含：

- 当日采购成功/失败次数摘要
- 多条采购单记录
- 当日无采购动作空态

#### 5.2 单条采购记录当前已展示内容

- 时间
- 采购单号
- 供应商 / 渠道名称
- 当前结果标签（成功 / 失败）
- 失败原因或执行说明
- 采购价
- 处理情况
- 处理人
- 确认号 / 失败归因
- 行级动作按钮

#### 5.3 当前已实现的行级动作

在采购记录层面，页面已经出现以下动作：

- 失败详情
- 重新采购
- 重发确认
- 人工补采
- 确认回写

说明：

- 这与旧文档中“详情页没有业务动作按钮”的说法不一致
- 当前 `index.html` 已明确是“查看 + 处理”复合页

### 8. 右侧状态摘要区

当前右侧第二块卡片为“状态摘要”。

#### 8.1 当前展示字段

- 系统状态
- 平台状态
- 售后状态
- 采购状态

#### 8.2 页面说明策略

页面已通过静态说明强调：

- 售后附加状态建议在售后记录中呈现
- 不与主状态区无限叠加堆叠

说明：

- 当前详情页与列表页一样，已经统一成四段式状态口径
- 采购状态是当前页面的重要主状态，不应再作为可选字段描述

当前详情页与抽屉详情页已统一使用以下状态色：

| 状态类别 | 颜色类名 | 典型状态 |
| --- | --- | --- |
| 待确认/待处理 | `tag-state-pending` | `PENDING`、平台待处理 |
| 处理中 | `tag-state-processing` | `BOOKING`、`CANCELING`、采购中 |
| 成功/履约完成 | `tag-state-success` | `BOOKED`、`CHECKED_IN`、`COMPLETED`、采购完成、采购单已确认 |
| 失败/拒绝/超时 | `tag-state-failed` | `BOOK_REJECTED`、`BOOK_FAILED`、`CANCEL_FAILED`、采购失败 |
| 履约异常定责 | `tag-state-no-show` | `NO_SHOW` |
| 已取消/关闭 | `tag-state-cancelled` | `CANCELLED`、采购已取消、采购单取消成功 |
| 售后中 | `tag-state-aftersale` | 已售后 |
| 说明性中性状态 | `tag-state-neutral` | 未进入售后、已处理、未采购、未知状态 |

## 二、抽屉版 side-drawer.html 的补充说明

虽然本次以 `index.html` 为主，但当前列表页实际打开的是 `side-drawer.html`，因此需要补充说明其差异。

### 1. 抽屉版定位

抽屉版更偏向：

- 在列表内快速查看一笔订单
- 上半部看主订单摘要
- 下半部按采购单逐条查看执行明细

### 2. 抽屉版当前已展示的典型字段

- 酒店名称 + ID / 地址 / 电话 tooltip
- 房型名称
- 入住人 / 联系人
- 取消规则
- 创建时间 / 支付时间 / 取消时间
- 采购单号
- 采购状态
- 房型信息
- 下单账号
- 开票类型
- 订单金额 + 价格明细
- 失败原因弹层
- 钟点房的入住时长 / 已选时段

### 3. 抽屉版当前结构特点

- 更偏“订单主信息 + 采购单详情卡片”
- 存在 `查看更多`、`价格明细` 等折叠能力
- 能更直接看到每一张采购单的状态与失败原因

## 三、当前文档与页面对齐后的关键修正

以下旧口径已被当前页面推翻，现统一修正：

1. 详情页不再是“左侧采购任务、右侧订单/采购信息”的三段式联动结构
2. 当前 `index.html` 已存在 SLA、风险提示、订单轨迹、底部固定操作栏
3. 当前 `index.html` 已存在“查看日志”入口
4. 当前 `index.html` 已存在接单、拒单、人工补采、发起售后、确认采购结果等动作
5. 详情页当前不仅查看结果，也查看采购过程摘要
6. 详情页状态口径已与列表页统一为四段式

## 四、与其他页面关系

### 1. 与列表页关系

- 列表页中的“订单详情”是当前详情页的主要入口
- 列表页当前通过抽屉方式承载详情内容
- 列表工作台负责找单和分流，详情页负责深入查看与处理

### 2. 与日志页关系

- 当前 `index.html` 已有“查看日志”入口
- 日志页负责更底层的链路排查与报文查看
- 详情页中的“订单轨迹”只承担摘要级过程展示

可以理解为：

- 详情页看“总览 + 处理”
- 日志页看“链路 + 报文”

## 五、后续维护建议

建议后续重点关注：

1. 主页面版与抽屉版是否继续并行维护
2. 采购记录区是否需要与日志页做更强联动
3. 底部动作的权限、显隐、状态依赖规则是否补细
4. 风险提示与 SLA 规则是否接入真实数据
5. 价格明细与支付补充信息的字段口径是否继续扩展
6. 钟点房、补采、失败重试等场景是否继续丰富

## 六、结论

当前酒店订单详情页已经演进为一个综合作业页，而不是旧文档中的静态详情页。

它当前最重要的特征是：

- 有首屏总览
- 有 SLA 与风险提示
- 有订单与入住信息
- 有按日聚合的采购尝试过程
- 有订单轨迹
- 有金额与状态摘要
- 有底部固定动作栏
- 有失败原因与补充信息弹层

后续所有相关文档都应以这一结构为基准继续更新，而不再沿用旧的左右三段式描述。
