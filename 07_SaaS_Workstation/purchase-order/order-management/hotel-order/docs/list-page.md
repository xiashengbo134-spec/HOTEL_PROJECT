# 酒店订单列表页设计

## 页面定位

酒店订单列表页是手工订单处理模块的主工作台，用于承接酒店订单的集中查询、任务分流、状态跟踪与快速处理。

该页面的核心目标包括：

- 统一承接各平台推送过来的酒店订单
- 让业务员优先处理“待处理”与“售后中”订单
- 支持通过多维筛选快速定位目标订单
- 支持从列表直接进入订单详情与订单日志
- 在列表层面完成接单、拒单等关键动作

## 文档说明

本页采用 **“当前已实现 / 已明确规划”** 的方式说明。

- 当前已实现：以你当前页面和确认口径为准
- 已明确规划：已确定要纳入设计，但可在后续迭代中继续补细节

## 一、当前已实现

### 1. 页面结构

当前列表页可理解为由以下几个区域组成：

1. 顶部筛选区
2. Tab 任务分类区
3. 订单列表区
4. 行级操作区
5. 顶部全局工具区（导出、导出记录等）

其中，本文重点描述列表工作台本身；导出及导出记录的弹窗和内部流程暂不展开。

### 2. Tab 定义

当前 Tab 定义已经明确，最终包括：

- 待处理
- 售后中
- 已接单
- 已拒单
- 已取消
- 全部订单

说明：

- `全部订单` 包含全部状态的订单
- `已拒单` 当前仅记录**手动拒单**
- 平台超时自动拒单的订单，不进入“已拒单”，而是进入“已取消”
- `FAILED` 等异常订单当前不单独成 Tab，默认在“全部订单”中可见，并由具体业务状态进行归类处理

### 3. 筛选区设计

#### 3.1 当前筛选项

当前列表页已确认包含以下筛选条件：

- 系统订单号
- 外部订单号
- 入住人/联系人
- 酒店名称/ID
- 时间筛选
- 城市
- 订单来源
- 平台订单状态

#### 3.2 各筛选项说明

##### 3.2.1 系统订单号

- 支持完整单号搜索
- 支持尾 6 位搜索
- 为独立输入框

##### 3.2.2 外部订单号

- 支持完整单号搜索
- 支持尾 6 位搜索
- 为独立输入框

说明：

- 系统订单号与外部订单号是**两个独立筛选框**
- 两者都支持尾 6 位检索

##### 3.2.3 入住人/联系人

该筛选项是聚合搜索框，命中范围包括：

- 入住人姓名
- 联系人姓名
- 手机号

说明：

- 页面文案可使用“入住人/联系人”
- 实际搜索口径覆盖姓名与手机号

##### 3.2.4 酒店名称/ID

该筛选项也是聚合搜索框，支持以下内容检索：

- 酒店名称
- 聚合 ID
- 平台 ID
- 销售 ID

说明：

- 文案层可统一显示为“酒店名称/ID”
- 实际匹配时需兼容名称与多个 ID 口径

##### 3.2.5 时间筛选

当前时间筛选框左侧支持下拉切换时间维度，当前已确认包括：

- 预订时间
  - 对应用户订单创建时间
- 入住时间
- 离店日期

说明：

- 时间选择命中当前下拉选中的时间类型
- 页面上属于一个时间筛选组件，不是三组并列日期筛选器

##### 3.2.6 城市

- 用于筛选酒店所在城市

##### 3.2.7 订单来源

订单来源不只是平台名称，还需要体现“该订单属于哪个平台中的哪个销售账号”。

例如：

- 飞猪平台下某个店铺账号
- 抖音平台下某个销售账号

说明：

- 订单来源字段本质上带有“平台 + 销售账号”双重识别含义

##### 3.2.8 平台订单状态

- 对应不同平台原生状态
- 支持多选
- 不直接等同于系统订单状态

### 4. Tab 任务视图与提醒

#### 4.1 当前视图作用

Tab 的作用不仅是切换视图，还承担以下业务职责：

- 按业务状态分类订单
- 指导业务员按优先级处理订单
- 决定行级按钮的展示规则

#### 4.2 红点提醒

当前已明确以下规则：

- “待处理”Tab：展示数字红点
- “售后中”Tab：展示数字红点
- 其他 Tab：默认不展示红点

红点数字用于表达当前待处理任务量与售后任务量。

### 5. 列表核心展示内容

当前列表页的展示核心不是单一表格字段，而是围绕“订单身份、用户行程、状态、金额、操作”几个维度进行信息组织。

#### 5.1 订单标识信息

建议/当前口径包括：

- 外部订单号
- 系统订单号
- 复制按钮
- 渠道 Icon
- 订单来源

说明：

- 外部订单号、系统订单号后均有复制能力
- 复制属于轻操作，目的是提高排查与沟通效率

#### 5.2 酒店与行程信息

建议按同一信息组展示：

- 酒店名称
- 房型名称
- 早餐信息
- 入住人
- 联系方式
- 入住日期
- 离店日期
- 间夜数

#### 5.3 金额信息

当前列表中**只展示支付金额与结算金额**。

但其展示逻辑存在业务背景：

- 销售账号存在“底价模式”和“卖价模式”配置
- 因此页面不直接完整平铺所有财务字段
- 列表层仅保留最关键的两项金额信息

当前展示方式：

- 支付金额
- 结算金额
- 鼠标悬浮查看完整财务信息

悬浮信息示例口径包括：

- 支付金额：`¥251.24`
- 结算金额：`¥224.61`
- 佣金：`¥26.63`
- 佣金比例：`10.60%`
- 退款金额：`-`
- 取消罚金：`-`
- 售卖模式：卖价 / 底价

说明：

- 列表页文档中应明确“支付金额、结算金额是摘要展示，完整财务项通过悬浮查看”
- 页面无需在列表层强行平铺佣金、退款金额、取消罚金等字段

#### 5.4 状态栏

当前列表状态栏由三部分组成：

- 系统订单状态
- 平台订单状态
- 售后状态

其中，售后状态不仅展示主状态，还可配合下方业务标签展示更具体的处理结果或当前动作，例如：

- 售后中
- 已处理
- 协商退款
- 已同意
- 已取消

说明：

- 状态展示属于复合状态区
- 文档中需明确三类状态不可混用
- 售后状态可包含“主状态 + 补充标签”的表达方式

### 6. 行级操作规则

#### 6.1 当前操作范围

当前列表行级操作已确认如下：

- 采购详情（实际页面名称应统一为“订单详情”入口）
- 接单
- 拒单
- 日志

此外还存在轻量操作：

- 复制外部订单号
- 复制系统订单号

#### 6.2 状态下的按钮显隐规则

| 业务状态 Tab | 订单详情 | 日志 | 接单 | 拒单 |
| --- | --- | --- | --- | --- |
| 待处理 | 显示 | 显示 | 显示 | 显示 |
| 售后中 | 显示 | 显示 | 不显示 | 不显示 |
| 已接单 | 显示 | 显示 | 不显示 | 不显示 |
| 已拒单 | 显示 | 显示 | 不显示 | 不显示 |
| 已取消 | 显示 | 显示 | 不显示 | 不显示 |
| 全部订单 | 显示 | 显示 | 按订单实际是否待处理决定 | 按订单实际是否待处理决定 |

说明：

- 只有待处理订单有接单、拒单操作
- 其他状态无接单、拒单操作
- 订单详情、日志为全局常驻操作

#### 6.3 接单 / 拒单交互

- 接单：点击后弹出二次确认弹窗
- 拒单：点击后弹出二次确认弹窗
- 当前两者均**不需要备注**
- 当前两者均**不需要填写拒单原因**

说明：

- 当前版本以确认弹窗为准
- 后续如业务需要扩展原因字段，可在后续版本补充，不写入当前已实现口径

### 7. SLA 倒计时规则

#### 7.1 当前口径

SLA 倒计时并非统一固定值，而是由不同平台返回不同处理规则决定。

例如：

- 飞猪平台：处理时限 120 分钟
- 抖音平台：处理时限 90 分钟

#### 7.2 计时起点

倒计时从**用户支付成功、平台将订单推送过来**开始计算。

说明：

- 这不是单纯按系统创建时间计算
- 也不是统一固定时长
- 页面应根据平台返回规则与起算时间动态展示剩余时长

#### 7.3 生效范围

- 仅对“待处理”订单生效

#### 7.4 超时处理规则

当前已明确：

- 超时后标记为“已超时”
- 系统自动拒单
- 订单进入“已取消”Tab
- 不进入“已拒单”Tab

原因：

- `已拒单` 当前只记录**手动拒单**
- 平台超时自动拒单归入取消口径处理

#### 7.5 提醒方式

- 后续计划增加声音提醒
- 当前版本暂不做声音提醒

## 三、与其他页面关系

### 1. 与详情页关系

- 列表中的“订单详情”操作进入详情页
- 详情页用于查看订单详情信息与采购任务详情

### 2. 与日志页关系

- 日志页不是独立菜单入口
- 当前仅支持从列表页单条订单的“日志”按钮进入
- 日志以侧边弹窗形式统一查看

## 五、结论

当前酒店订单列表页已经不是单纯的查询页，而是一个面向业务员的订单工作台。

它的核心特征是：

- 以 Tab 管理业务任务视图
- 以多维筛选支持快速检索
- 以复合状态表达订单处理上下文
- 以支付金额 / 结算金额摘要支撑快速财务判断
- 以接单 / 拒单 / 日志 / 详情形成闭环操作链路

后续如果继续补页面原型说明，可以在此基础上再进一步补：

- 空态
- 无权限态
- 加载态
- 超时高亮样式
- 状态标签颜色体系
- 列表字段排版建议
