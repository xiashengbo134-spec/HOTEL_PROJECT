# 酒店订单状态流转说明

## 文档目的
本文件用于定义酒店订单模块中的状态体系与流转规则，统一产品、设计、前端、后端对状态含义的理解。

本模块中的状态分为三层：

- 业务状态：面向工作台视图与业务分流
- 系统订单状态：面向系统内部订单生命周期管理
- 平台状态：面向外部 OTA 平台状态同步与映射

---

## 一、状态分层说明

### 1. 业务状态（父状态）
业务状态主要用于列表页 Tab 分类、任务分流、红点提醒、操作权限控制。

建议定义如下：

| 业务状态 | 说明 | 典型场景 |
| --- | --- | --- |
| 待处理 | 新进入工作台、等待人工接单或处理的订单 | 新单待确认 |
| 售后中 | 订单已进入取消、退款、异常处理、补单等售后流程 | 取消中、退款中、异常处理中 |
| 已接单 | 订单已被业务员接手，处于正常跟进或已完成采购 | 已接单、采购中、已预订 |
| 已拒单 | 订单被人工拒绝接单 | 无法承接、价格异常、资源不足 |
| 已取消 | 订单已完成取消或关闭 | 平台取消、系统取消、售后取消完成 |
| 全部订单 | 聚合视图，不属于实际状态值 | 查询视图 |

### 2. 系统订单状态（子状态）
系统订单状态用于精确描述订单当前生命周期阶段，是驱动详情页展示、列表状态、日志记录的核心状态。

建议定义如下：

| 系统状态 | 中文说明 | 状态类型 | 说明 |
| --- | --- | --- | --- |
| `PENDING` | 待确认 | 初始态 | 订单已进入系统，等待人工处理 |
| `BOOKING` | 预订中 | 中间态 | 已开始采购或预订流程 |
| `BOOKED` | 已预订 | 成功态 | 已完成采购并确认成功 |
| `CHECKED_IN` | 已入住 | 履约完成态 | 用户已入住 |
| `CANCELING` | 取消中 | 中间态 | 已发起取消，等待平台/渠道确认 |
| `CANCELED` | 已取消 | 完结态 | 订单已取消完成 |
| `FAILED` | 失败 | 异常态 | 采购失败、下单失败、同步失败等 |
| `REJECTED` | 已拒绝 | 完结态 | 订单被人工拒单 |
| `CLOSED` | 已关闭 | 完结态 | 非取消场景下的关闭态，可按实际系统决定是否保留 |

> 说明：`REJECTED` 与 `CLOSED` 是否作为独立系统状态，可根据后端现有状态模型调整；若暂未落地，可先保留在文档层。

### 3. 平台状态
平台状态是指飞猪、抖音、携程等外部平台回传的订单状态。由于各平台命名不统一，前台不直接以平台原始值作为主判断依据，而是统一映射到内部系统状态和业务状态。

平台状态的处理原则：

- 原始平台状态保留，用于排查与日志追踪
- 前台展示使用统一后的平台状态文案
- 平台状态需映射到系统订单状态
- 系统订单状态再参与业务状态计算

## 二、业务状态与系统状态映射建议

### 1. 映射原则
- 列表 Tab 依据业务状态展示
- 详情页核心标签以系统状态为主
- 当系统状态与售后状态同时存在时，优先判断是否进入“售后中”
- 平台状态仅作为外部同步参考，不直接替代系统状态

### 2. 建议映射表

| 业务状态 | 系统状态范围 | 补充条件说明 |
| --- | --- | --- |
| 待处理 | `PENDING` | 尚未接单，且未进入售后 |
| 已接单 | `BOOKING` / `BOOKED` / `CHECKED_IN` | 已被接单，且未进入售后 |
| 售后中 | `CANCELING` / `FAILED` / 其他异常处理中状态 | 已进入取消、退款、补单、售后处理 |
| 已拒单 | `REJECTED` | 人工明确拒单 |
| 已取消 | `CANCELED` / `CLOSED` | 已完成取消或关闭 |

> 若订单处于 `FAILED`，但业务定义上仍需回到“待处理”由人工二次判断，也可作为特例单独定义。

## 三、典型状态流转路径

### 1. 正常下单链路

`PENDING` → `BOOKING` → `BOOKED` → `CHECKED_IN`

说明：
- 订单进入系统后处于待确认
- 业务员接单或系统触发采购后进入预订中
- 采购成功后进入已预订
- 入住日履约后进入已入住

### 2. 取消链路

`PENDING / BOOKING / BOOKED` → `CANCELING` → `CANCELED`

说明：
- 订单任一处理中阶段都可能发起取消
- 发起后进入取消中
- 平台或渠道确认取消成功后进入已取消

### 3. 失败与售后链路

`BOOKING` → `FAILED` → 售后处理 / 人工补单 / 再采购

说明：
- 采购失败、支付失败、同步失败等进入失败态
- 失败后可触发售后任务
- 也可能进入人工补单、重新采购等补救流程
- 如果补救成功，可重新回到 `BOOKING` 或 `BOOKED`

### 4. 人工拒单链路

`PENDING` → `REJECTED`

说明：
- 仅待处理订单允许人工拒单
- 拒单后应写入拒单原因与日志
- 拒单后不可再执行接单操作

## 五、SLA 倒计时规则

### 1. 生效范围
仅对“待处理”业务状态订单生效。

### 2. 目的
- 提醒业务员优先处理高时效订单
- 降低超时未处理风险
- 为自动拒单、违规统计提供基础依据

### 3. 展示建议
- 展示位置：订单来源区域或订单主标识附近
- 展示形式：倒计时文案，如 `59分07秒`
- 颜色规则：
  - `> 30 分钟`：灰色/绿色
  - `< 15 分钟`：橙色
  - `< 5 分钟`：红色

### 4. 超时处理建议
倒计时归零后，可根据业务规则执行：

- 自动拒单
- 标记超时未处理
- 进入异常工作台
- 写入系统日志并触发提醒

## 六、详情页状态展示规则

### 1. 双状态并存
详情页需同时展示：

- 主订单状态
- 采购执行状态

两者不可混用。

### 2. 展示原则

| 状态类型 | 说明 | 建议样式 |
| --- | --- | --- |
| 主订单状态 | 表示系统内部订单总状态 | 主状态 Tag |
| 平台状态 | 表示平台同步态 | 次级说明或标签 |
| 采购状态 | 表示具体采购任务执行态 | 独立 Tag |
| 售后状态 | 表示取消/退款/补单等处理态 | 辅助 Tag |

### 3. 颜色建议

| 状态类别 | 建议颜色 |
| --- | --- |
| 成功/完成 | 绿色 |
| 处理中/中间态 | 橙色或蓝色 |
| 失败/异常 | 红色 |
| 失效/关闭 | 灰色 |

由于平台状态差异较大，建议后续单独维护平台状态字典。当前可先定义统一映射原则：

| 平台侧含义 | 建议映射系统状态 | 建议映射业务状态 |

| 已下单/处理中 | `BOOKING` | 已接单 |
| 预订成功 | `BOOKED` | 已接单 |

| 取消处理中 | `CANCELING` | 售后中 |
| 已取消 | `CANCELED` | 已取消 |
| 下单失败/履约失败 | `FAILED` | 售后中 / 待人工判断 |

## 八、日志记录要求

以下状态动作建议必须记录日志：

- 接单
- 拒单
- 发起采购
- 采购成功
- 采购失败
- 发起取消
- 取消成功
- 创建售后任务
- 售后处理完成
- 状态同步失败
- 超时自动处理

日志字段建议至少包含：

- 订单号
- 外部单号
- 采购任务号
- 原状态
- 新状态
- 操作类型
- 操作人
- 操作时间
- 备注/原因
- traceId

## 十、结论

酒店订单模块的状态设计应坚持以下原则：

- 列表看业务状态，详情看系统状态，排查看日志状态
- 平台状态只做输入，不直接作为前台唯一判断依据
- 业务操作显隐必须与状态严格绑定
- 超时、取消、失败、售后等异常流必须可回溯

该文档可作为后续接口字段定义、前端状态展示和后端状态机实现的共同参考基线。
