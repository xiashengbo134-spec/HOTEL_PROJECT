# 酒店订单状态流转说明

## 文档目的
本文件用于定义酒店订单模块中的状态体系与流转规则，统一产品、设计、前端、后端对状态含义的理解。

本模块中的状态分为三层：

- 业务状态：面向工作台视图与业务分流
- 系统订单状态：面向系统内部订单生命周期管理
- 平台状态：面向外部 OTA 平台状态同步与映射

---

## 一、状态分层说明

### 1. 业务状态（父状态）
业务状态主要用于列表页 Tab 分类、任务分流、红点提醒、操作权限控制。

当前按业务规则统一为：

| 业务状态 | 判定口径 | 典型场景 |
| --- | --- | --- |
| 待处理 | 系统订单状态 = `PENDING`、`BOOKING` | 新单待确认、已推送采购但仍需关注 |
| 售后中 | 订单售后状态 = 已售后（`aftersale_state=1`） | 平台出现售后退款、取消协商、异常退款 |
| 已接单 | 系统订单状态 = `BOOKED`、`CHECKED_IN`、`COMPLETED`、`NO_SHOW` | 已预订成功、已入住、已完成、未到店扣款 |
| 已拒单 | 系统订单状态 = `BOOK_REJECTED`、`BOOK_FAILED` | 平台拒单、系统兜底失败 |
| 已取消 | 系统订单状态 = `CANCELLED`、`CANCELING`、`CANCEL_FAILED` | 平台已取消、系统取消处理中或取消失败 |
| 全部订单 | 聚合视图，不属于实际状态值 | 查询视图 |

### 2. 系统订单状态（子状态）
系统订单状态用于精确描述订单当前生命周期阶段，是驱动详情页展示、列表状态、日志记录的核心状态。

当前按业务口径统一为：

| 系统状态 | 中文说明 | 状态类型 | 判定说明 |
| --- | --- | --- | --- |
| `PENDING` | 待确认 | 初始态 | 订单初始状态 |
| `BOOKING` | 预订中 | 中间态 | 交付任务状态 = 已推送，已发起采购 |
| `BOOKED` | 已预订 | 成功态 | 存在采购订单状态 = 预订成功 |
| `CHECKED_IN` | 已入住 | 履约态 | 存在采购单状态 = 已入住 |
| `CANCELING` | 取消中 | 中间态 | 存在采购单状态 = 取消中 |
| `CANCELLED` | 已取消 | 完结态 | 平台订单已取消 |
| `CANCEL_FAILED` | 取消失败 | 异常完结态 | 存在采购单状态 = 取消失败、取消被拒且不存在取消中 |
| `BOOK_REJECTED` | 预订被拒 | 失败态 | 平台订单被拒 |
| `BOOK_FAILED` | 预订失败 | 失败态 | 兜底失败 |
| `NO_SHOW` | 未到店 | 履约异常态 | 存在采购单状态 = 未入住，取消方被扣款 |
| `COMPLETED` | 已完成 | 完结态 | 存在采购单状态 = 已完成 |

### 3. 平台状态
平台状态是指飞猪、抖音、携程等外部平台回传的订单状态。由于各平台命名不统一，前台不直接以平台原始值作为主判断依据，而是统一映射到内部系统状态和业务状态。

平台状态的处理原则：

- 原始平台状态保留，用于排查与日志追踪
- 前台展示使用统一后的平台状态文案
- 平台状态需映射到系统订单状态
- 系统订单状态再参与业务状态计算

---

## 二、业务状态与系统状态映射建议

### 1. 映射原则
- 列表 Tab 依据业务状态展示
- 详情页核心标签以系统状态为主
- 当系统状态与售后状态同时存在时，优先判断是否进入“售后中”
- 平台状态仅作为外部同步参考，不直接替代系统状态

### 2. 建议映射表

| 业务状态 | 系统状态范围 | 补充条件说明 |
| --- | --- | --- |
| 待处理 | `PENDING` / `BOOKING` | 尚未完结，仍在待确认或采购发起阶段 |
| 售后中 | 不依赖系统状态单独判断 | `aftersale_state=1` 时优先进入售后中 |
| 已接单 | `BOOKED` / `CHECKED_IN` / `COMPLETED` / `NO_SHOW` | 正常履约、已完成或未到店扣款 |
| 已拒单 | `BOOK_REJECTED` / `BOOK_FAILED` | 平台拒单或系统兜底失败 |
| 已取消 | `CANCELLED` / `CANCELING` / `CANCEL_FAILED` | 平台订单已取消，系统展示为取消链路 |

## 三、典型状态流转路径

### 1. 正常下单链路

`PENDING` → `BOOKING` → `BOOKED` → `CHECKED_IN`

说明：
- 订单进入系统后处于待确认
- 业务员接单或系统触发采购后进入预订中
- 采购成功后进入已预订
- 入住日履约后进入已入住

### 2. 取消链路

`PENDING / BOOKING / BOOKED` → `CANCELING` → `CANCELED`

说明：
- 订单任一处理中阶段都可能发起取消
- 发起后进入取消中
- 平台或渠道确认取消成功后进入已取消

### 3. 失败与售后链路

`BOOKING` → `FAILED` → 售后处理 / 人工补单 / 再采购

说明：
- 采购失败、支付失败、同步失败等进入失败态
- 失败后可触发售后任务
- 也可能进入人工补单、重新采购等补救流程
- 如果补救成功，可重新回到 `BOOKING` 或 `BOOKED`

### 4. 人工拒单链路

`PENDING` → `REJECTED`

说明：
- 仅待处理订单允许人工拒单
- 拒单后应写入拒单原因与日志
- 拒单后不可再执行接单操作

## 五、SLA 倒计时规则

### 1. 生效范围
仅对“待处理”业务状态订单生效。

### 2. 目的
- 提醒业务员优先处理高时效订单
- 降低超时未处理风险
- 为自动拒单、违规统计提供基础依据

### 3. 展示建议
- 展示位置：订单来源区域或订单主标识附近
- 展示形式：倒计时文案，如 `59分07秒`
- 颜色规则：
  - `> 30 分钟`：灰色/绿色
  - `< 15 分钟`：橙色
  - `< 5 分钟`：红色

### 4. 超时处理建议
倒计时归零后，可根据业务规则执行：

- 自动拒单
- 标记超时未处理
- 进入异常工作台
- 写入系统日志并触发提醒

## 六、详情页状态展示规则

### 1. 四段状态并存
详情页与列表页保持一致，需同时展示：

- 系统状态
- 平台状态
- 采购状态
- 售后状态

四者语义不同，不可混用。

### 2. 展示原则

| 状态类型 | 说明 | 展示要求 |
| --- | --- | --- |
| 系统状态 | 表示系统内部订单总状态 | 作为主状态标签展示 |
| 平台状态 | 表示 OTA / 平台同步状态 | 使用统一映射后的平台文案展示 |
| 采购状态 | 表示采购任务执行状态 | 独立标签展示，不与系统状态混用 |
| 售后状态 | 表示退款/取消/补单等售后处理态 | 辅助标签展示 |

### 3. 颜色规范

| 颜色类别 | 适用状态 |
| --- | --- |
| 蓝色 `tag-state-pending` | `PENDING`、平台待处理、待确认类状态 |
| 橙色 `tag-state-processing` | `BOOKING`、`CANCELING`、采购中等处理中状态 |
| 绿色 `tag-state-success` | `BOOKED`、`CHECKED_IN`、`COMPLETED`、采购完成、平台已确认 |
| 红色 `tag-state-failed` | `BOOK_REJECTED`、`BOOK_FAILED`、`CANCEL_FAILED`、采购失败 |
| 紫色 `tag-state-no-show` | `NO_SHOW` 等履约异常但已定责状态 |
| 灰色 `tag-state-cancelled` | `CANCELLED`、采购已取消、关闭类状态 |
| 红色强调 `tag-state-aftersale` | 已售后、售后处理中 |
| 中性色 `tag-state-neutral` | 未进入售后、已处理、未采购、说明性状态 |

## 八、日志记录要求

以下状态动作建议必须记录日志：

- 接单
- 拒单
- 发起采购
- 采购成功
- 采购失败
- 发起取消
- 取消成功
- 创建售后任务
- 售后处理完成
- 状态同步失败
- 超时自动处理

日志字段建议至少包含：

- 订单号
- 外部单号
- 采购任务号
- 原状态
- 新状态
- 操作类型
- 操作人
- 操作时间
- 备注/原因
- traceId

## 十、结论

酒店订单模块的状态设计应坚持以下原则：

- 列表看业务状态，详情看系统状态，排查看日志状态
- 平台状态只做输入，不直接作为前台唯一判断依据
- 业务操作显隐必须与状态严格绑定
- 超时、取消、失败、售后等异常流必须可回溯

该文档可作为后续接口字段定义、前端状态展示和后端状态机实现的共同参考基线。
