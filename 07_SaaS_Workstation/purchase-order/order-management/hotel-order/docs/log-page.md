# 酒店订单日志页设计

## 页面定位

酒店订单日志页用于查看单笔订单的全链路处理过程，承担“解释订单为什么会变成现在这样”的职责。

结合当前 `pages/log/index.html` 原型，日志页已经形成一个较完整的**抽屉式排查控制台**，支持：

- 订单摘要快速确认
- 阶段视图回放链路
- 业务视图聚焦分类
- 订单 / 采购范围切换
- 报文抽屉查看与复制

---

## 一、入口与形态

### 1. 当前入口

当前日志页主要入口仍然是：

- 列表页单条订单“日志”按钮

同时需要补充说明：

- 当前详情页主原型 `pages/detail/index.html` 中已经出现“查看日志”按钮
- 因此“详情页完全没有日志入口”的旧口径已不再适用于当前全部原型
- 但实际工作流中，列表页仍然是最主要入口

### 2. 当前展示形态

- 以右侧抽屉方式打开
- 抽屉宽度与列表页中的原型约为 1400px 级别
- 页面本体为 `pages/log/index.html`

说明：

- 当前日志页仍然不是独立菜单页面
- 它是订单工作台中的深度排查能力

## 二、当前页面已实现

### 1. 页面整体结构

当前日志页可拆为四个层次：

1. 顶部订单摘要区
2. Tab 与查看范围控制区
3. 主体日志区
   - 阶段日志视图
   - 业务日志视图
4. 日志报文抽屉

### 2. 顶部订单摘要区

当前页面顶部不是简单标题，而是**摘要芯片区**。

#### 2.1 当前已展示字段

根据现有页面，顶部摘要区已实现：

- 系统订单号
- 外部订单号
- 酒店名称
- 房型名称 - 早餐
- 入离时间
- 间夜 / RP

#### 2.2 当前已实现的辅助能力

- 酒店名称旁支持查看酒店聚合 ID
- 房型名称旁支持查看房型聚合 ID
- 订单号类字段采用截断 + 单行摘要布局

说明：

- 这比旧文档里“只有标题、酒店名、订单号”的描述更完整
- 当前日志页首屏已经能够独立确认订单上下文

### 3. Tab 与范围控制区

当前摘要区下方已实现两个核心控制能力：

#### 3.1 日志视图 Tab

- 阶段日志
- 业务日志

#### 3.2 查看范围切换

- 只看订单
- 只看采购单

说明：

- 这是当前页面已经落地的重要过滤能力
- 旧文档里把它放在“后续规划”已不准确，应改为当前已实现

## 三、阶段日志视图

### 1. 视图定位

阶段日志是当前页面默认主视图，用于按流程阶段回放订单处理链路。

它更适合回答：

- 哪个阶段开始异常
- 某次采购是在什么阶段失败
- 售后、取消、退款是按什么顺序发生的
- 某个采购任务号对应的动作链路是什么

### 2. 当前页面结构

当前阶段视图不是“左侧节点 + 右侧表格”的老结构，而是**单栏手风琴式阶段列表**：

- 每个阶段为一行摘要
- 可展开查看该阶段合并的多条日志
- 每行直接显示时间、内容、phase、结果、操作人、详情按钮

#### 2.1 当前摘要行字段

- 日期
- 时间
- 主内容 `content`
- 阶段码 `phase`
- 执行结果 `result`
- 操作人 `operator`
- 详情按钮

#### 2.2 当前已实现的阶段合并能力

- 相同阶段日志可合并展示
- 页面会提示“相同阶段合并，共 X 条”
- 可一键展开 / 收起
- 页面还支持“全部展开 / 全部收起”

#### 2.3 展开后的明细行

展开后会展示同阶段的隐藏日志，字段包括：

- 时间
- 内容
- 阶段码
- 执行结果
- 操作人
- 单条日志按钮

说明：

- 当前阶段视图已把“聚合摘要 + 明细展开”做成一个统一界面
- 不再依赖旧版“点击左侧节点刷新右侧区”的交互模型

### 3. 当前典型阶段

当前 mock 中已覆盖的阶段包括：

- `VALIDATE`
- `ORDER_CREATE`
- `PURCHASE_INQUIRY`
- `PURCHASE_TASK_CREATE`
- `PURCHASE_TASK_PAY`
- `PURCHASE_TASK_QUERY`
- `PURCHASE_TASK_CANCEL`
- `PURCHASE_TASK_STATE_CHANGE`
- `ORDER_AFTERSALE_CREATE`
- `ORDER_CANCEL`
- `ORDER_REFUND`
- `ORDER_AFTERSALE_RECEIVE`
- `ORDER_AFTERSALE_PROCESSED`

### 4. 当前交互特征

- 默认进入阶段日志
- 支持整页滚动查看
- 同阶段节点可展开收起
- 每个阶段都可直达“详情”查看报文
- 展开后的单条记录也可打开“日志”报文

## 四、业务日志视图

### 1. 视图定位

业务日志视图按业务类别聚焦日志，适合快速看某一类动作是否异常。

### 2. 当前布局

业务视图当前实现为左右结构：

- 左侧：业务分类列表
- 右侧：该分类下的结果明细列表

### 3. 当前分类

当前已实现的业务分类包括：

- 试单
- 下单
- 采购
- 取消（页面数据实际映射自“售后/取消”）

每个分类当前还展示：

- 状态（成功 / 失败）
- 日志条数

### 4. 右侧明细区

当前业务视图右侧明细区已实现：

- 总记录数
- 时间
- 结果

说明：

- 业务视图当前更强调“快速扫结果”
- 不像阶段视图那样逐段展开过程

---

## 五、日志报文抽屉

### 1. 当前形态

当前页面已实现独立的“日志报文详情”抽屉：

- 从右侧滑出
- 宽度约 `960px`
- 标题：日志报文详情

### 2. 当前能力

- 显示日志对应 JSON 报文
- 支持复制报文
- 支持收起抽屉
- 深色代码块风格显示报文内容

说明：

- 当前不是在表格内内联展开 JSON
- 而是统一收敛到一个更大的二级抽屉中查看
- 这比旧文档中的“表格行展开”描述更符合当前原型

## 六、当前聚合与判定逻辑

### 1. 数据来源

当前原型通过前端内嵌 `rawLogs` 数据模拟底层日志。

### 2. 日志范围判定

页面当前按 phase 判断 scope：

- `PURCHASE*` 相关日志归入采购范围
- 其他日志归入订单范围

这也是“只看订单 / 只看采购单”切换的基础。

### 3. 阶段聚合

当前阶段视图会对日志进行阶段级聚合，并支持：

- 合并同类阶段
- 保留主日志
- 展示隐藏日志数量
- 单独保留 tail log

### 4. 状态判断

页面当前会根据结果文本和 `isError` 来控制：

- 错误高亮
- 成功 / 失败的视觉区分
- 分类状态展示

### 5. 采购任务场景覆盖

当前 mock 已覆盖以下采购相关问题：

- 采购任务创建
- 支付设备不可用
- 采购轮询
- 采购取消
- 状态变更
- 人工重新发起补单

这与详情页中的采购过程查看形成呼应。

## 七、当前页面与旧文档的关键差异修正

以下旧口径已不再准确，现统一修正：

1. 顶部摘要不再只有简单标题和订单号，而是多芯片摘要区
2. 页面已实现“只看订单 / 只看采购单”切换
3. 阶段视图不是左导航 + 右明细，而是单栏手风琴聚合列表
4. 报文查看不是表格内展开，而是独立 JSON 报文抽屉
5. 详情页主原型中已出现“查看日志”按钮，不能再绝对写死“详情页无日志入口”

---

## 八、当前 mock 场景说明

当前原型模拟了一笔订单从试单、下单、采购、取消、售后到人工补单的完整过程，覆盖了：

- 试单成功
- 下单成功
- 采购询价
- 采购任务创建
- 采购任务支付失败
- 采购任务轮询
- 采购取消
- 订单取消
- 售后任务创建 / 领取 / 处理
- 订单退款
- 人工补单任务生成

该 mock 场景的价值在于：

- 能解释为什么一笔订单会存在多次采购动作
- 能说明列表页的业务状态、详情页的采购过程、日志页的底层链路之间如何对应

## 九、与其他页面关系

### 1. 与列表页关系

- 列表页是当前日志页的主要入口页
- 列表页告诉用户“现在是什么状态”
- 日志页解释“为什么会变成这个状态”

### 2. 与详情页关系

- 详情页中会展示采购过程摘要与订单轨迹
- 日志页则提供更底层的阶段明细、结果明细与报文

可以概括为：

- 详情页看摘要过程
- 日志页看底层过程

## 十、后续维护建议

建议后续重点维护以下内容：

1. 顶部摘要字段是否继续扩展
2. 业务分类字典是否变化
3. 阶段聚合规则是否与采购任务模型联动调整
4. 范围切换是否扩展为更多 scope
5. 报文抽屉是否增加格式化、搜索、结构化视图
6. 是否补充导出、仅看异常、时间筛选等增强能力

---

## 十一、结论

当前酒店订单日志页已经形成较完整的抽屉式排查控制台。

它当前最重要的特征是：

- 顶部多芯片订单摘要
- 阶段日志 / 业务日志双视图
- 订单范围 / 采购范围切换
- 手风琴式阶段聚合
- 独立 JSON 报文抽屉
- 与列表页、详情页形成“状态 - 摘要 - 过程”的联动

后续相关文档应以这一实现结构为基线继续维护，而不再沿用旧版“左右分栏表格式日志”的描述。
