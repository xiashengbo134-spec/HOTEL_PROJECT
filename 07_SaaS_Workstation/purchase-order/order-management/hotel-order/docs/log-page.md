# 酒店订单日志页设计

## 页面定位

酒店订单日志页用于展示单笔订单的全链路追踪日志，帮助业务员、订单管理员、售后人员从时序和业务两个维度回溯订单的处理过程。

该页面的核心价值包括：

- 排查订单从平台进入系统后的完整处理链路
- 还原主订单、采购任务、售后动作之间的先后关系
- 定位采购失败、取消异常、退款处理、售后流转等问题
- 支撑列表页和订单详情页之外的深度追踪能力

与列表页、详情页相比，日志页不是用于“处理订单”，而是用于**解释订单为什么会变成现在这样**。

## 文档说明

本页采用 **“当前已实现 / 已明确规划”** 的方式说明。

- 当前已实现：以当前原型与已确认页面口径为准
- 已明确规划：已确定方向，但可在后续迭代中继续补充

## 一、页面角色与入口边界

### 1. 页面角色

日志页不是独立菜单入口，而是酒店订单工作台中的订单排查能力。

它的职责边界可理解为：

- **列表页**：看订单、筛订单、处理订单
- **详情页**：看订单详情、看采购任务、看采购信息
- **日志页**：看链路、看过程、看原因、看报文

因此，日志页是对列表页与详情页的补充，不替代两者。

### 2. 页面入口

当前已确认：

- 日志页仅从**订单列表页单条订单的“日志”按钮**进入
- 详情页当前**没有**日志入口
- 也没有单独的“采购日志”入口

### 3. 展示形态

当前日志页以**侧边抽屉**方式打开，而不是独立主页面。

说明：

- 这种方式更符合“从列表排查某一单”的使用路径
- 用户无需离开订单工作台即可查看日志
- 查看完成后可快速返回列表继续处理下一单

## 二、当前已实现

### 1. 当前已落地原型

- 原型文件：`../pages/log/index.html`
- 当前版本：1400px 宽度的抽屉式日志控制台
- 展示方式：经典实心时间轴 + 阶段日志 / 业务日志双视图
- 原型特征：高信息密度、强调时序追踪、适合订单排查场景

### 2. 页面整体结构

当前日志页可拆分为以下三个区域：

1. 顶部摘要区
2. 左侧日志导航区
3. 右侧日志明细区

### 3. 顶部摘要区

当前原型顶部以抽屉标题形式展示：

- 页面标题：订单日志
- 酒店名称
- 订单号

结合现有列表页与详情页口径，顶部摘要区承担的作用是：

- 让用户确认当前查看的是哪一笔订单
- 避免在批量排查时看错单
- 在高密度日志中始终保留订单识别信息

#### 3.1 当前已展示字段

- 页面标题：订单日志
- 酒店名称：如 `杭州西湖万豪酒店`
- 订单号：如 `HTL202604228899`

#### 3.2 结合现有业务建议补充的摘要字段

后续建议补充以下字段，以便与列表页、详情页形成更完整的对照：

- 外部订单号
- 平台订单号 / 渠道订单号
- 当前系统订单状态
- 平台订单状态
- 订单来源
- 入住日期 / 离店日期
- traceId

说明：

- 这些字段并不是当前必须全部落地
- 但从排查效率角度看，越接近列表与详情页的核心识别字段，日志页越好用

### 4. 左侧日志导航区

左侧区域承担“按维度聚合日志并导航”的作用，当前通过 Tabs 支持两类视图：

- 阶段日志
- 业务日志

这两种视图分别对应两种排查思路：

- **阶段日志**：适合看顺序、看节点、看流程是否断掉
- **业务日志**：适合看某类操作是否集中出问题，如采购、售后、取消

## 三、阶段日志视图

### 1. 视图定位

阶段日志视图用于以时间轴方式回放订单的关键生命周期节点，是当前日志页的主视图。

它适合回答的问题包括：

- 订单是否成功创建
- 是否生成过采购任务
- 采购任务进行了几次
- 失败发生在采购的哪个环节
- 是否发生过取消、退款、售后处理
- 当前订单为什么落在某个业务状态中

### 2. 当前展示形式

左侧采用时间轴形式展示业务流转过程，当前实现特征包括：

- 使用实心状态图标区分成功、失败、进行中
- 每个节点展示时间、标题、日志条数、操作人、状态
- 采购任务类节点支持展示关联任务信息
- 点击节点后联动右侧明细区

### 3. 当前阶段聚合对象

当前页面中的阶段聚合对象包括但不限于：

- `VALIDATE`（可定检查）
- `ORDER_CREATE`（创建订单）
- `PURCHASE_INQUIRY`（采购询价）
- `PURCHASE_TASK_CREATE`（采购任务创建）
- `PURCHASE_TASK_PAY`（采购任务支付）
- `PURCHASE_TASK_QUERY`（采购任务轮询）
- `PURCHASE_TASK_CANCEL`（采购任务取消）
- `PURCHASE_TASK_STATE_CHANGE`（采购任务状态变更）
- `ORDER_AFTERSALE_CREATE`（创建售后任务）
- `ORDER_CANCEL`（订单取消）
- `ORDER_REFUND`（订单退款）
- `ORDER_AFTERSALE_RECEIVE`（领取售后任务）
- `ORDER_AFTERSALE_PROCESSED`（处理售后任务）

### 4. 与订单详情页的关系

阶段日志视图与订单详情页中的“采购任务区”存在天然关联，但两者职责不同：

- **详情页左侧采购任务区**：展示当前订单下有哪些采购任务
- **日志页阶段视图**：展示这些采购任务是如何被创建、执行、失败、取消、重试的

因此，详情页告诉用户“有什么任务”，日志页告诉用户“任务经历了什么”。

### 5. 适用场景

阶段日志特别适合以下场景：

- 列表页看到订单处于售后中，但不清楚是因为什么进入售后
- 详情页看到存在多个采购任务，想知道这些任务是如何演变出来的
- 需要判断采购失败后是否发生了重试采购、人工补单或取消原任务
- 需要回放取消、退款、售后处理的先后顺序

## 四、业务日志视图

### 1. 视图定位

业务日志视图用于从业务分类角度查看日志，而不是按时间轴节点拆解。

它适合回答的问题包括：

- 当前订单的采购类日志是否集中报错
- 售后/取消过程中有哪些关键动作
- 某一类业务有没有重复执行、异常执行

### 2. 当前业务分类

当前按业务类型聚合为：

- 试单
- 下单
- 采购
- 售后/取消

每个分类展示对应日志数量，点击后右侧展示该业务分类下的全部日志明细。

### 3. 与列表页状态栏的关系

业务日志视图与列表页中的复合状态栏可以互相印证：

- 列表页能看到系统状态、平台状态、售后状态的结果
- 日志页业务分类能解释这些状态变化背后发生了哪些业务动作

例如：

- 列表页显示“售后中 + 协商退款”
- 日志页可进一步查看是否创建了售后任务、是否有人领取、是否已处理、是否进入退款流程

### 4. 适用场景

业务日志特别适合：

- 快速聚焦采购异常
- 快速聚焦售后/取消动作
- 无需逐节点浏览时间轴，直接看某类日志集合

---

## 五、右侧日志明细区

### 1. 视图定位

右侧明细区用于承接左侧选中的阶段节点或业务分类，并以表格形式呈现底层日志记录。

该区域是日志页中信息最细的一层，用于回答：

- 某个节点究竟执行了什么
- 谁执行了该动作
- 执行结果是什么
- 是否有报文可供进一步排查

### 2. 当前表格字段

当前表格字段包括：

- 精确时间
- 阶段（`phase` / `phaseName`）
- 执行内容
- 执行结果
- 操作人
- 展开查看 JSON 报文

### 3. 展示规则

其中：

- 精确时间拆分为日期与时间两行展示
- 阶段列同时展示 `phaseName` 与 `phase`
- 错误日志使用红色高亮
- 成功结果使用绿色高亮
- 表格行支持展开查看 JSON 报文
- 支持复制报文内容

### 4. 与详情页采购信息区的关系

详情页采购信息区展示的是“当前任务的结果信息”，而日志明细区展示的是“生成这些结果过程中的明细记录”。

例如：

- 详情页可看到某采购任务状态为失败
- 日志页明细可继续看到失败发生在支付、轮询还是取消阶段
- 详情页可看到采购价格、采购账号等结果字段
- 日志页可帮助定位这些字段是如何写入、变更、失败重试的

## 六、当前原型交互说明

### 1. 阶段日志交互

- 默认进入“阶段日志”Tab
- 默认自动选中第一个阶段节点
- 点击左侧任意阶段节点，右侧刷新该节点对应的明细日志
- 选中状态以高亮边框与背景色表示

### 2. 业务日志交互

- 点击“业务日志”Tab 后，左侧切换为业务分类列表
- 点击某个业务分类后，右侧显示该分类下全部日志明细
- 当前默认选中分类为“采购”

### 3. 报文交互

- 表格行支持展开
- 若该条日志存在 `json` 字段，则显示 JSON 报文区域
- 支持点击“复制内容”按钮复制报文
- 若不存在报文，则展示“该日志节点无附属 JSON 报文”提示

### 4. 抽屉交互意义

由于当前日志页通过列表页侧边抽屉打开，因此在交互上具备以下特点：

- 不打断列表工作流
- 适合连续查看多笔订单日志
- 更符合“排查后即关闭”的轻量分析场景

## 七、当前页面中的聚合逻辑说明

### 1. 底层日志数据

页面当前通过 `rawLogs` 数组承载底层日志数据。

### 2. 阶段聚合规则

当前原型将日志分为两类：

- 宏观节点：非 `PURCHASE_TASK*` 日志
- 批次节点：`PURCHASE_TASK*` 相关日志

### 3. 批次节点识别

采购任务相关日志会尝试从 `content` 或 `result` 中提取采购任务单号，用于同一采购任务批次聚合。

这一点与详情页左侧采购任务区的业务理解高度一致：

- 详情页按任务卡片看采购任务
- 日志页按任务号把采购相关日志聚合成可排查的批次节点

### 4. 状态判定规则

当前状态判定规则包括：

- 存在错误日志：标记为失败
- 批次最后阶段仍为查询类：标记为进行中
- 无错误且流程完成：标记为成功
- 被后续宏观节点切断但无错误：仍标记为成功

### 5. 操作人展示规则

节点操作人按聚合日志去重处理，优先展示非“系统”操作人，否则展示“系统”。

- 整体页面宽度：1400px 抽屉
- 风格关键词：高密度、经典、业务排查导向
- 时间轴节点：实心状态图标
- 状态颜色：

- 视觉上优先保证“可读、可追踪、可比对”

## 九、当前 mock 数据场景说明

当前原型模拟了一条酒店订单从试单、下单、采购、采购失败重试、取消、退款、售后领取、售后处理到人工补单的完整链路。

模拟场景覆盖：

- 试单成功
- 下单成功
- 采购询价
- 采购任务创建
- 采购支付失败（无可用支付设备）
- 采购轮询
- 采购取消
- 订单取消
- 售后任务创建 / 领取 / 处理
- 订单退款
- 人工补单任务生成

该 mock 场景与当前详情页的多采购任务模式是匹配的，能够用于说明：

- 为什么一个订单会出现多次采购任务
- 为什么订单最终会进入售后或取消
- 为什么列表页状态与采购结果之间可能不是简单一一对应关系

## 十二、与其他页面关系

### 1. 与列表页关系

- 来源页面：`../pages/list/`
- 触发方式：列表页单行“日志”按钮打开日志抽屉
- 典型使用场景：客服排查、运营追单、采购排障、售后回溯

日志页的核心价值在于：

- 列表页告诉你“这单现在是什么状态”
- 日志页告诉你“这单为什么会变成这个状态”

### 2. 与详情页关系

- 关联页面：`../pages/detail/`
- 详情页展示订单信息与采购信息结果
- 日志页展示这些结果背后的处理过程

可以理解为：

- 详情页看“结果态”
- 日志页看“过程态”

两者结合后，才能完整理解一笔酒店订单的全链路处理过程。

## 十四、结论

当前酒店订单日志页本质上是订单工作台中的排查控制台。

它的核心特征是：

- 仅从订单列表页进入
- 以抽屉形式承接快速排查需求
- 通过阶段视图回放订单与采购任务的时序过程
- 通过业务视图聚焦试单、下单、采购、售后/取消等分类日志
- 与订单详情页形成“结果页 + 过程页”的互补关系

后续如果继续细化原型说明，可以在此基础上继续补：

- 顶部摘要字段规范
- 节点状态色与标签规则
- 业务分类字典
- 报文结构展示规范
- 异常日志优先级与筛选规则
- 抽屉模式与独立页模式的切换原则
