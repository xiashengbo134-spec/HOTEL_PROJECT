# 酒店订单模块概览

## 模块定位

酒店订单模块用于承接手工订单处理场景下的酒店订单管理，覆盖：

- 订单进入工作台后的筛选与分流
- 接单 / 拒单等人工处理动作
- 采购过程与采购结果核查
- 状态变化、售后链路与日志排查

从当前页面原型看，这已经不是一个简单查询模块，而是一个围绕**任务处理、采购履约、过程排查**构建的业务工作台。

## 菜单归属

- 一级菜单：采购/订单
- 二级菜单：订单管理
- 三级菜单：酒店订单

## 当前页面能力总览

当前模块由三个核心页面能力组成：

- 列表页 `list-page.md`
- 详情页 `detail-page.md`
- 日志页 `log-page.md`

但基于最新原型，三者关系需要按下面方式理解。

---

## 一、列表页：订单工作台

列表页是当前模块主入口，也是任务操作最集中的页面。

### 当前已体现的能力

- 多维筛选
- Tab 任务分流
- 待处理 SLA 倒计时
- 四段式状态展示
  - 系统状态
  - 平台状态
  - 采购状态
  - 售后状态
- 支付金额 / 结算金额摘要
- 接单 / 拒单
- 订单详情抽屉
- 日志抽屉
- 导出 / 导出记录入口

### 当前页面特征

- 不是表格，而是卡片式订单工作台
- 待处理 Tab 支持按剩余处理时间排序
- 钟点房场景已直接在卡片中兼容展示

因此，列表页主要回答的问题是：

> 现在有哪些订单需要处理，我该先处理哪一单？

---

## 二、详情页：订单详情作业页

详情页页面名称统一为：**订单详情**。

基于当前最新 `pages/detail/index.html` 原型，详情页已经不是旧版文档中的“左侧采购任务、右侧主单信息”三段式结构，而是一个更完整的综合作业页。

### 当前已体现的能力

- 订单总览卡片
- SLA 与风险提示
- 订单与入住信息查看
- 按日期聚合的采购单过程查看
- 订单轨迹回放
- 金额与结算摘要
- 状态摘要
- 底部固定操作栏
- 失败原因 / 支付补充 / 价格明细等弹层

### 当前页面特征

- 既看结果，也看过程摘要
- 既支持核对，也已经出现处理动作
- 当前主原型中已经出现“查看日志”入口

因此，详情页主要回答的问题是：

> 这笔订单现在整体是什么情况，采购过程经历了什么，还需要我做什么？

### 需要补充说明的双形态

当前详情能力实际存在两种原型：

1. `pages/detail/index.html`
   - 独立详情页/弹层式详情页原型
2. `pages/detail/side-drawer.html`
   - 列表页当前实际使用的抽屉版详情页原型

两者都属于“订单详情”能力，但展现重点不同：

- `index.html` 更偏总览与作业
- `side-drawer.html` 更偏列表内快速查看

---

## 三、日志页：订单日志排查台

日志页用于查看单笔订单的全链路日志。

### 当前已体现的能力

- 顶部多芯片订单摘要
- 阶段日志 / 业务日志双视图
- 只看订单 / 只看采购单范围切换
- 阶段手风琴聚合展开
- JSON 报文抽屉查看与复制

### 当前页面特征

- 仍以抽屉方式承载
- 重点是过程回放、异常定位、报文查看
- 与详情页中的“订单轨迹”形成摘要层 / 明细层关系

因此，日志页主要回答的问题是：

> 这笔订单为什么会变成现在这样，具体在哪个阶段、哪条日志出了问题？

---

## 四、三个页面的职责边界

结合最新页面实现，建议按下表理解：

| 页面 | 核心问题 | 当前职责 |
| --- | --- | --- |
| 列表页 | 现在有哪些单要处理？ | 找单、筛单、分流、快速处理 |
| 详情页 | 这单整体情况如何、还要做什么？ | 看总览、看采购过程摘要、执行局部动作 |
| 日志页 | 这单具体为什么变成现在这样？ | 看阶段、看分类、看报文、排查原因 |

可以概括为：

- 列表页看**任务态**
- 详情页看**总览态 / 作业态**
- 日志页看**过程态 / 排查态**

---

## 五、状态体系概览

当前模块页面中实际已经形成较稳定的状态分层：

### 1. 业务状态

用于列表页 Tab 分流：

- 待处理
- 售后中
- 已接单
- 已拒单
- 已取消

### 2. 页面主状态

当前列表页与详情页都已经显式展示：

- 系统状态
- 平台状态
- 采购状态
- 售后状态

这意味着页面层文档不应再只按“系统状态 + 平台状态 + 售后状态”三段式描述。

### 3. 详细状态映射

更细的枚举与映射关系仍建议放在：

- `state-machine.md`

## 六、关键业务规则概览

### 1. 接单 / 拒单规则

- 仅待处理订单可接单、拒单
- 需二次确认
- 当前不要求填写备注或拒单原因

### 2. SLA 规则

- 仅待处理订单生效
- 列表页直接展示倒计时
- 详情页首屏也展示剩余处理时间
- 超时后自动拒单并进入已取消

### 3. 采购过程规则

当前原型已明确支持以下过程理解：

- 多次采购尝试
- 失败重试
- 人工补采
- 渠道切换
- 按日期聚合采购过程查看

### 4. 日志查看规则

- 当前主要从列表页进入
- 当前详情页主原型中也已出现日志入口
- 日志页承担深度排查，不承担主业务处理

## 八、当前已实现结论

基于最新页面原型，当前模块已经明确实现了以下协同关系：

- 列表页负责承接任务与快速动作
- 详情页负责承接总览、采购过程摘要与局部处理
- 日志页负责承接链路、报文与异常排查

这三者已经可以较完整地支撑酒店手工订单处理中的：

- 查询
- 分流
- 处理
- 核对
- 排查

## 十、结论

当前酒店订单模块已经形成清晰的三层协同：

- 列表页负责**找单与处理**
- 详情页负责**总览与作业**
- 日志页负责**追踪与排查**

后续若页面结构再次变化，应优先更新本概览文档，再同步更新各分页面文档，避免页面设计口径继续沿用旧版描述。
