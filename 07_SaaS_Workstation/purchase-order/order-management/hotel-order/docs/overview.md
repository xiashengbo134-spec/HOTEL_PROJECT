# 酒店订单模块概览

## 模块定位

酒店订单模块用于承接**手工订单处理场景下的酒店订单管理**，覆盖从订单进入工作台、人工处理、采购任务追踪，到日志排查、状态流转、售后回溯的完整链路。

该模块不是单纯的订单查询模块，而是一个围绕“订单处理 + 采购执行 + 链路排查”展开的业务工作台，主要服务对象包括：

- 订单业务员
- 订单管理员
- 售后处理人员
- 采购协同人员

模块目标包括：

- 快速识别待处理与售后中订单
- 通过多维检索快速定位目标订单
- 在详情页中核对主订单信息与采购执行信息
- 通过日志页回溯状态变化、采购过程与异常原因
- 为后续接口、状态机、字段字典沉淀统一业务基线

## 菜单归属

- 一级菜单：采购/订单
- 二级菜单：订单管理
- 三级菜单：酒店订单

## 功能架构

当前酒店订单模块可理解为三个核心层次：

1. **订单工作台层（列表页）**
   - 承接订单查询、筛选、状态分流、接单/拒单等快速处理动作
2. **订单详情作业层（详情页）**
   - 承接主订单信息查看、采购任务切换、采购执行信息对照
3. **状态与追踪支撑层（日志 + 状态机）**
   - 承接订单状态流转、采购过程追踪、售后链路回放与异常排查

## 页面结构

当前模块由三个核心页面能力组成：

- 列表页 `list-page.md`
- 详情页 `detail-page.md`
- 日志页 `log-page.md`

三者关系如下：

### 1. 列表页：订单工作台

列表页是模块主入口，承担“看单、筛单、分流、处理”的职责。

核心作用：

- 按 Tab 管理订单任务视图
- 对订单进行多维筛选
- 展示订单核心信息、复合状态和金额摘要
- 提供接单、拒单、订单详情、日志等操作入口

当前核心 Tab 已明确为：

- 待处理
- 售后中
- 已接单
- 已拒单
- 已取消
- 全部订单

其中：

- “待处理”和“售后中”需具备数量提醒能力
- `全部订单` 包含全部状态订单
- `已拒单` 仅记录手动拒单
- 超时自动拒单的订单进入 `已取消`

### 2. 详情页：订单详情

详情页页面名称统一为：**订单详情**。

该页面不是纯订单信息页，而是由“主订单信息 + 采购任务 + 当前采购信息”共同构成的综合详情页。

当前结构已明确为：

- 左侧：采购任务区
- 右侧上方：订单信息区
- 右侧下方：采购信息区

核心作用：

- 查看主订单信息
- 查看订单关联的采购任务列表
- 切换不同采购任务
- 对照采购结果与主订单需求
- 辅助判断失败、重试、拆单、补单等采购场景

### 3. 日志页：订单日志抽屉

日志页用于查看单笔订单的全链路日志，当前仅作为列表页中的扩展排查能力存在。

核心作用：

- 回放订单从创建到售后/取消的全过程
- 排查采购任务创建、执行、失败、取消、重试等链路
- 按阶段或按业务分类查看日志明细
- 查看底层报文与执行结果

当前已确认：

- 日志页不是独立菜单页面
- 日志页仅从列表页单条订单“日志”按钮进入
- 详情页当前没有日志入口
- 日志页以侧边抽屉方式展示

## 页面职责边界

为了避免页面职责混淆，当前模块建议按以下边界理解：

| 页面 | 核心问题 | 页面职责 |
| --- | --- | --- |
| 列表页 | 现在有哪些订单需要处理？ | 查询、筛选、分流、快速操作 |
| 详情页 | 这笔订单和采购任务具体是什么情况？ | 查看主订单与采购结果 |
| 日志页 | 这笔订单为什么会变成现在这样？ | 查看过程、排查原因、回放链路 |

可以概括为：

- 列表页看**任务态**
- 详情页看**结果态**
- 日志页看**过程态**

## 状态体系概览

当前模块的状态体系分为三层：

### 1. 业务状态

主要用于列表页 Tab 视图和任务分流，包括：

- 待处理
- 售后中
- 已接单
- 已拒单
- 已取消

### 2. 系统订单状态

用于表达订单生命周期，例如：

- `PENDING`
- `BOOKING`
- `BOOKED`
- `CHECKED_IN`
- `CANCELING`
- `CANCELED`
- `FAILED`

### 3. 平台订单状态

用于承接不同 OTA 平台的原生状态，并映射到内部系统状态和业务状态。

说明：

- 列表页中当前会同时展示系统订单状态、平台订单状态、售后状态
- 状态的详细映射关系在 `state-machine.md` 中单独维护

## 关键业务规则概览

### 1. 接单 / 拒单规则

- 只有待处理订单才有接单、拒单操作
- 接单、拒单都需要二次确认弹窗
- 当前不需要填写备注或拒单原因

### 2. SLA 倒计时规则

- 仅对待处理订单生效
- 倒计时按不同平台规则计算
- 从用户支付成功且平台推单开始计算
- 超时后自动拒单，并进入已取消

### 3. 多采购任务规则

一个订单下不一定一定存在采购任务。

当前已确认以下场景会导致一个订单存在多个采购任务：

- 失败重试采购
- 分单采购
- 人工补单
- 多渠道比价后重新采购

### 4. 日志入口规则

- 当前日志入口仅存在于列表页
- 详情页当前没有日志入口
- 日志页用于解释状态变化和处理过程，不承载业务处理动作
