# hotel-order

对应三级菜单：酒店订单。

本模块承接**手工订单处理场景下的酒店订单管理**，核心覆盖从订单工作台检索、订单处理，到采购执行详情追踪、日志回溯的一整套作业链路。结合目前已完成的页面规划与原型实现，本目录重点管理以下三类页面：

- 订单工作台（列表页）
- 采购作业详情（详情页）
- 订单/采购日志（日志页）

该 README 作为模块级入口文档，用于统一说明当前模块的功能架构、页面职责、状态体系、关键交互及后续文档沉淀方向。

## 模块定位

酒店订单模块是采购订单管理中的核心业务子模块，主要服务于业务员/采购员对酒店订单进行人工处理与过程跟踪，目标包括：

- 快速识别待处理订单并进行高优处理
- 对订单进行多维检索、筛选、分类查看
- 在详情页中同时对比主订单信息与实际采购执行信息
- 跟踪采购任务、状态变化、售后过程与异常处理记录
- 为后续的接口设计、状态流转、日志留痕提供统一业务基线

## 功能架构设计

整个手工订单处理模块可拆分为三个核心层次：

1. **工作台层（列表）**：负责订单集中查询、任务分流、批量查看与快速操作
2. **详情作业层（详情）**：负责单笔订单的主单信息、采购任务、采购执行信息查看
3. **底层状态引擎（状态流转）**：负责业务状态、系统状态、平台状态之间的关联与驱动

## 页面范围

- 列表页 `pages/list/`
- 详情页 `pages/detail/`
- 日志页 `pages/log/`

## 页面设计说明

### 1. 订单工作台（列表页）

列表页是酒店订单模块的主入口，承担“任务收口 + 状态分流 + 快速处理”的核心职责。

#### 1.1 多维检索模块

支持按以下条件进行精准或组合查询：

- 外部单号
- 系统单号
  - 支持输入完整单号
  - 支持尾 6 位检索
- 预订时间范围
- 入住日期范围
- 城市
- 订单来源
- 平台订单状态
- 其他后续扩展业务条件

其中，订单编号类检索应兼顾：

- 精准搜索
- 模糊搜索
- 多条件组合筛选

#### 1.2 任务分类 Tab（视图管理）

列表页按业务任务视角划分为以下 Tab：

- 待处理
- 售后中
- 已接单
- 已拒单
- 已取消
- 全部订单

说明：

- “待处理”与“售后中”Tab 需支持**数字红点提醒**
- 红点数字用于提示当前高优任务量，便于业务员优先处理
- Tab 本质上是业务视图切换，不仅影响展示，也影响行级操作权限与按钮显隐

#### 1.3 列表核心展示内容

每条订单需尽量完整表达“订单身份 + 酒店信息 + 财务信息 + 状态信息 + 时间信息 + 操作入口”，建议包含以下列：

- 订单标识
  - 外部单号
  - 系统单号
  - 渠道 Icon
  - 订单来源
- 酒店与房型信息
  - 酒店名称
  - 房型名称
  - 是否含早
- 入住人与行程信息
  - 入住人
  - 联系方式
  - 入住日期
  - 离店日期
  - 间夜数
- 财务信息
  - 客户支付金额
  - 采购结算金额/底价
  - 卖价 vs 底价对比
- 复合状态栏
  - 系统状态
  - 平台状态
  - 售后状态
- 时间轴信息
  - 预订时间
  - 付款时间
  - 取消时间
- 操作指令区
  - 采购详情
  - 接单
  - 拒单
  - 日志
  - 导出

#### 1.4 列表页关键交互规则

##### Tab 切换与红点提醒

- “待处理”和“售后中”Tab 需展示实时数字提醒
- 红点数量用于表达当前任务积压情况
- 数量刷新应与列表数据口径保持一致

##### 倒计时警报机制（SLA）

仅针对“待处理”状态订单生效：

- 展示位置：可展示在订单来源图标下方或订单识别区域附近
- 展示内容：动态倒计时，例如 `59分07秒`
- 颜色分级建议：
  - `> 30 分钟`：灰色/绿色，表示相对安全
  - `< 15 分钟`：橙色，表示接近超时
  - `< 5 分钟`：红色，表示高风险
- 高风险阶段可结合声音提醒或更强视觉警示
- 若倒计时归零仍未处理，系统应进入超时预案，例如：
  - 自动拒单
  - 标记超时
  - 记录违规/异常处理日志

##### 操作按钮显隐与鉴权规则

- `接单 / 拒单`
  - 仅在“待处理”Tab 下展示且可点击
  - 在“已接单”“已取消”等非待处理状态下应隐藏或置灰
- `采购详情 / 日志`
  - 全局常驻
  - 任意状态下均可查看
- `导出`
  - 视角色权限与列表筛选结果开放
  - 后续可在接口文档中补充导出范围与字段定义

## 2. 采购作业详情（详情页）

详情页用于承接列表页的下钻操作，是业务员查看订单基准信息、采购任务与采购执行信息的核心作业页面。

### 2.1 页面结构

详情页建议采用左右结构：

- 左侧：关联任务侧边栏（Task Panel）
- 右侧上方：主订单信息（Master Order Info）
- 右侧下方：采购执行单（Purchase Execution Info）

### 2.2 左侧：关联任务侧边栏

用于展示该订单下属的采购任务列表，建议采用卡片式布局。

任务卡片建议包含：

- 任务状态（如：已停止、进行中、已完成等）
- 创建时间
- 关联单号
- 任务标识信息

交互要求：

- 若一个订单存在多次采购、重试采购或分单采购，应支持展示多个任务卡片
- 点击不同任务卡片后，右侧采购执行信息需联动刷新
- 当前选中任务需有明确高亮态

### 2.3 右侧上方：主订单信息

该区域展示系统接收到的客户订单基准信息，是业务判断的主参照。

建议包含：

- 酒店基础信息
- 客户支付金额
  - 需高亮展示，例如红色金额 `¥196.92`
- 入住人信息
- 联系方式
- 房型与行程信息
- 入住/离店日期
- 取消规则
- 主订单状态标签

设计重点：

- 金额信息应突出，便于与采购底价直接比对
- 订单基础信息需保证阅读层级清晰，避免关键信息被弱化

### 2.4 右侧下方：采购执行单

该区域展示业务员/系统实际去外部渠道采购后的执行信息。

建议包含：

- 采购渠道信息
  - 例如：携程、PTG 等
- 采购渠道单号
- 采购渠道状态
  - 例如：取消中
- 采购底价
  - 需高亮展示，例如红色金额 `¥181.19`
- 对应采购房型信息
- 对应入住人信息
- 实际采购账号信息
  - 例如：`CTRIP_TRIP_***`
- 采购备注、异常说明或补充字段
- 采购日志入口

设计重点：

- 采购底价与主单售价需形成上下视觉对照，帮助业务员快速判断利润与倒挂风险
- 主单状态与采购状态应分开展示，避免混淆“平台侧状态”和“采购侧状态”

### 2.5 详情页关键交互规范

#### 信息层级视觉对比

- 客户支付金额与采购底价都应采用醒目红色字体展示
- 两组金额建议在视觉上对齐，以便一眼完成成本/售价核对
- 若存在利润异常、倒挂、金额不一致等情况，可增加辅助提示样式

#### 状态对比展示

- 主单状态与采购单状态分别使用独立 Tag 展示
- 建议颜色规则：
  - 橙色：处理中/中间态
  - 绿色：已完成/已成功
  - 灰色：已失效/已关闭
  - 红色：失败/高风险异常

#### 卡片化任务切换

- 左侧任务卡片支持切换
- 切换后右侧采购渠道详情联动刷新
- 若无采购任务，应展示空态说明，而不是空白页面

## 3. 状态管理引擎

状态体系是整个酒店订单模块的底层驱动逻辑，需要明确区分“业务状态”“系统订单状态”“平台状态”三层含义。

### 3.1 业务状态（父状态）

面向工作台视图与业务分流，建议包含：

- 待处理
- 售后中
- 已接单
- 已拒单
- 已取消

这部分状态主要用于：

- 列表 Tab 分类
- 高优任务识别
- 行级操作控制
- 工作量统计与提醒

### 3.2 系统订单状态（子状态 / 核心驱动）

系统内部状态用于精确表达订单生命周期，建议至少覆盖：

- `PENDING`：待确认
- `BOOKING`：预订中
- `BOOKED`：已预订
- `CHECKED_IN`：已入住
- `CANCELING`：取消中
- `CANCELED`：已取消
- `FAILED`：失败（含各类失败场景）

说明：

- 系统状态是详情页、日志页与状态流转判断的核心依据
- 业务父状态通常由系统状态、售后状态、平台反馈等组合映射得出

### 3.3 平台状态映射

由于订单可能来自飞猪、抖音等外部 OTA 平台，因此需要对平台状态进行接收、同步与映射。

主要目标包括：

- 屏蔽不同平台状态命名差异
- 将平台状态统一映射到内部系统状态/业务状态
- 支持状态同步失败、延迟、冲突等异常场景记录

后续建议在 `docs/` 中进一步沉淀：

- 平台状态字典
- 平台状态 -> 系统状态映射规则
- 系统状态 -> 业务视图状态映射规则
- 售后状态与取消状态的边界说明

## 页面间交互关系

- 日志页不作为独立菜单入口
- 日志页由酒店订单列表中单条订单的“日志”操作按钮触发
- 详情页由订单编号或“采购详情”按钮进入
- 详情页底部可提供“采购日志”查阅入口
- 当详情页存在多个采购任务时，由左侧任务卡片驱动右侧详情内容切换
